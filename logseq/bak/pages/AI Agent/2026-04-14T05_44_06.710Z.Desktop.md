	- ## Harness
		- ### Guidance
			- #### Loop Pattern
				- one task/feature per iteration, two phases — generate then backpressure (tests/linters/compilers)
				- speed of feedback > perfect correctness
				- each session ends with: git commit, updated progress, no major bugs
				- `while :; do cat PROMPT.md | claude-code ; done`
				- > [Ralph] trust LLM to decide what's most important next; commit after tests pass, tag on zero errors, increment patch version from 0.0.0
			- #### Agent Roles
				- separate agents by concern — initializer (setup), coder (incremental progress), planner (expand specs), evaluator (validate output)
				- single generalist agents consistently underperform multi-agent systems with clear role boundaries
				- evaluators overcome self-praise bias
				- agent-to-agent review can replace most human review over time
				- > [Anthropic] **initializer agent** sets up environment on first run, **coding agent** makes incremental progress each session, leaving clear artifacts for the next. [quickstart](https://github.com/anthropics/claude-quickstarts/tree/main/autonomous-coding)
				- > [Anthropic] for complex tasks: **planner** expands prompts into detailed specs, **generator** implements, **evaluator** validates — separating evaluation from generation overcomes self-praise bias
			- #### Documentation as Infrastructure
				- keep AGENT.md short (~100 lines) as a table of contents, not an encyclopedia
				- a giant file crowds out the task, rots instantly, and becomes non-guidance
				- knowledge base lives in structured `docs/` directory as system of record
				- enforce freshness mechanically with linters and "doc-gardening" agents
				- anything not in the repo effectively doesn't exist for the agent (Slack threads, Google Docs, tacit knowledge)
				- > [Ralph] essential file stack loaded every loop: `@PROMPT.md` (instructions), `@fix_plan.md` (prioritized todos), `@specs/*` (one per file), `@AGENT.md` (how to build/run/test). Let LLM self-update AGENT.md with learnings; document bugs in fix_plan.md even if out of scope.
				- > [OpenAI] structured `docs/` as system of record. Design docs catalogued with verification status + core beliefs. [Architecture doc](https://matklad.github.io/2021/02/06/ARCHITECTURE.md.html) provides top-level map. Quality doc grades each domain/layer, tracking gaps over time.
			- #### Planning
				- write specs before code
				- have planner agent expand minimal prompts into comprehensive specs
				- use prioritized plans (`fix_plan.md` / execution plans) as first-class versioned artifacts checked into the repo
				- regenerate frequently — stale plans are worse than no plans
				- > [Ralph] begin with long conversations establishing requirements, generate formal specs before implementation. Wrong code usually means wrong specs.
				- > [OpenAI] ephemeral plans for small changes; complex work in [execution plans](https://cookbook.openai.com/articles/codex_exec_plans) with progress + decision logs checked into repo. Active plans, completed plans, and tech debt all versioned and co-located.
			- #### Sprint Contracts
				- before implementation, agents negotiate explicit contracts defining scope, testable success criteria, implementation approach, and constraints
				- prevents spec drift and ensures evaluators grade against agreed expectations
			- #### Architecture
				- agents work best in rigid, predictable structures — this is an early prerequisite, not a late optimization
				- enforce strict boundaries (dependency directions, layer separation) via custom linters and structural tests
				- prefer "boring" technologies — composable, stable APIs, well-represented in training data
				- write custom lint error messages that inject remediation instructions into agent context
				- enforce invariants (e.g. validate at boundaries), not implementations
				- > [OpenAI] each business domain divided into fixed layers (Types → Config → Repo → Service → Runtime → UI) with strictly validated dependency directions. Cross-cutting concerns enter through single explicit interface (Providers). Constraints enforced by custom linters (agent-generated) and structural tests.
			- #### Application Legibility
				- make the app itself legible to agents — bootable per worktree
				- UI driveable via Chrome DevTools/Puppeteer MCP (DOM snapshots, screenshots, navigation)
				- logs/metrics/traces queryable via local observability stack (LogQL, PromQL)
				- lets agents reproduce bugs, validate fixes, and reason about runtime behavior directly
				- > [Ralph] loop back is everything — always look for opportunities to feed output back into the LLM for evaluation. Add logging, compile and inspect output, etc.
				- > [OpenAI] ephemeral observability stack per worktree (logs, metrics, traces), torn down after task completion. Agents query with LogQL/PromQL. Single runs can work 6+ hours autonomously.
			- #### Testing & Quality Gates
				- test immediately after each unit
				- use browser automation for end-to-end verification — agents identify and fix bugs not obvious from code alone
				- use static analysis for dynamic languages
				- set minimum thresholds per quality dimension; if any falls below, the sprint fails with detailed feedback
				- never let agents remove or edit tests
				- > [Ralph] include docstrings explaining *why* tests exist — future loops lose context. For dynamic languages, wire in static analysers ([Pyrefly](https://pyrefly.org/), [Dialyzer](https://www.erlang.org/doc/apps/dialyzer/dialyzer.html)) or "you will run into a bonfire of outcomes."
				  ```markdown
				  After implementing functionality or resolving problems, run the tests for that unit of code that was improved. If functionality is missing then it's your job to add it as per the application specifications. Think hard.
				  ```
			- #### Context Management
				- compaction preserves continuity but doesn't give a clean slate — models can develop "context anxiety" and prematurely wrap up
				- context resets need good handoff artifacts
				- use git history + progress logs to supplement; they provide clarity compaction cannot guarantee
				- budget ~170k tokens per loop
				- > [Anthropic] Claude Agent SDK has compaction for long tasks without exhausting context window, but structured artifacts and explicit procedures are still needed to supplement.
			- #### Subagents & Isolation
				- parallelize reads/searches freely, restrict builds/tests to single agent
				- give each agent an isolated environment (worktree + ephemeral observability stack, torn down after task completion)
				- > [Ralph] up to 500 subagents for reads/searches/writes; primary context acts as scheduler, not executor. ~170k token budget per loop.
			- #### Evaluation Loops
				- for creative/subjective work, use generator + evaluator agent pairs with explicit grading criteria
				- 5-15 iterations per generation, improvement plateaus but middle iterations can beat final ones
				- tune evaluators by reading their logs and fixing where judgment diverged from expectations
				- > [Anthropic] four grading criteria for design: **Design quality** (coherent whole, distinct mood), **Originality** (custom decisions vs template/AI patterns), **Craft** (typography, spacing, color, contrast), **Functionality** (usability independent of aesthetics). Later iterations generally better, but middle ones sometimes preferred.
			- #### Prevent Drift & Entropy
				- explicitly instruct "search before creating", "no placeholders", "full implementations"
				- agents replicate existing patterns — even bad ones
				- encode "golden principles" and run recurring cleanup agents as garbage collection
				- technical debt compounds; pay it down continuously in small increments, not painful bursts
				- use feature list with pass/fail status to prevent premature completion
				- > [Ralph] LLMs chase compilation rewards — instruct "DO NOT IMPLEMENT PLACEHOLDER... WE WANT FULL IMPLEMENTATIONS". Search for TODOs, minimal implementations, placeholders. Always "search codebase first, don't assume not implemented" — ripgrep can yield false negatives.
			- #### Recovery
				- expect broken states
				- options: `git reset --hard`, rescue prompts, or switch models
				- when agent struggles, treat it as a signal — identify what capability is missing (tools, guardrails, docs) and feed it back into the repo
				- > [Ralph] try different models (e.g. Gemini) if one gets stuck in error loops

				- With this in mind, I wrote four grading criteria that I gave to both the generator and evaluator agents in their prompts:
				- **Design quality:** Does the design feel like a coherent whole rather than a collection of parts? Strong work here means the colors, typography, layout, imagery, and other details combine to create a distinct mood and identity.
				- **Originality:** Is there evidence of custom decisions, or is this template layouts, library defaults, and AI-generated patterns? A human designer should recognize deliberate creative choices. Unmodified stock components—or telltale signs of AI generation like purple gradients over white cards—fail here.
				- **Craft:** Technical execution: typography hierarchy, spacing consistency, color harmony, contrast ratios. This is a competence check rather than a creativity check. Most reasonable implementations do fine here by default; failing means broken fundamentals.
				- **Functionality:** Usability independent of aesthetics. Can users understand what the interface does, find primary actions, and complete tasks without guessing?
				- I ran 5 to 15 iterations per generation, with each iteration typically pushing the generator in a more distinctive direction as it responded to the evaluator's critique.
				- Across runs, the evaluator's assessments improved over iterations before plateauing, with headroom still remaining. Some generations refined incrementally. Others took sharp aesthetic turns between iterations.
				- Later implementations tended to be better as a whole, but I regularly saw cases where I preferred a middle iteration over the last one.
